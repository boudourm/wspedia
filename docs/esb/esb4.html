Architecture d’un ESB

L'ESB est représenté comme un canal séparé des autres services à travers lequel les informations et les instructions circulent :

×

Le consommateur accède au registre de services regroupant les contrats de services pour en prendre connaissance, puis, il appelle le service demandé en envoyant une requête via les adaptateurs de protocoles ou les connecteurs directs, ces derniers permettent à l’ESB de supporter divers protocoles de communication dans son dialogue avec les consommateurs et les services.

A la réception d’un appel, le connecteur transforme la requête en un message interne à l’ESB, cette approche est appelée « messagerie asynchrone ». Ce message est ensuite transporté sur le bus de transport des messages.

Le bus détermine la localisation du service métier à appeler et assure le routage du message vers ce dernier d’une manière sécurisée et suivant des règles définies dans le moteur de règles de routage.

Le service en question peut être local dans le container SCA1 ou distant publié par des systèmes externes, sa description peut être accédée via le registre de services. Une fois arrivé, le moteur de transformation de l’ESB convertit le message reçu au format attendu par le service (ex. : le format attendu par le WS SOAP est SOAP). Le message est transmis ultérieurement à l’adaptateur approprié (ex. : adaptateur HTTP) qui, à son tour, invoque le service.

Les ESB se concentrent sur les fonctions d'interconnexion et de médiation, et s'appuient pour cela sur un ensemble de standards parmi lesquels :

  • Les Web Services pour gérer les communications synchrones.
  • Le langage XML pour définir les formats des messages.
